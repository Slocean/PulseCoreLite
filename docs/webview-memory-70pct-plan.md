# WebView2 内存降低 70% 优化方案

目标：将 WebView2 相关占用从约 **600MB** 降至 **≤180MB（-70%）**，并尽量保持功能不变。

> 说明：当前采样中 WebView2 进程数稳定 6~7 个，平均约 90MB/进程，总和约 630MB。要达到 70% 降幅，必须从结构层面减少 WebView2 进程数，而不是仅靠小幅优化。

---

## 结论先行
- **仅优化前端页面与缓存不足以达标**。要稳定降到 ≤180MB，必须 **减少 WebView2 进程数**。
- 可控达标路径只有：**合并为单 WebView（主窗口复用为任务栏模式）** 或 **用原生渲染替换任务栏 WebView**。

---

## 方案 A（强制达标，推荐）
### 单 WebView 模式：主窗口复用为任务栏窗口

**核心思路**：只保留一个 WebView 窗口，通过“模式切换”改变窗口的位置、尺寸和布局。

**效果**：
- WebView2 进程数从 6~7 个压缩到 2~3 个（主进程 + 少量辅助进程）
- 有概率实现 **≥70%** 降幅

**代价**：
- 主窗口与任务栏窗口 **不能同时显示**（需要切换逻辑）

**改造落点**：
- `src/stores/windowStore.ts` 里的 `ensureMainWindow` 与 `openTaskbarMonitor` 逻辑
- `src/pages/index.vue` 增加“taskbar 内嵌模式”布局

**实施步骤**（高层）：
1. 统一只创建 `main` WebView。
2. 新增“任务栏模式”：
   - 调整窗口尺寸与位置（使用当前任务栏定位逻辑）。
   - 渲染 taskbar 组件到主窗口的某一模式。
3. 任务栏显示/隐藏时，切换主窗口模式与位置。

---

## 方案 B（中等可行，达标不保证）
### taskbar 原生渲染 + main 保留 WebView

**核心思路**：
- 保留 `main` WebView
- taskbar 窗口用 Rust 原生渲染（winit + skia/direct2d），避免 WebView2

**效果**：
- WebView2 进程数减少 1~2 个
- 预期 **降幅约 30~50%**，达标不保证

**代价**：
- 引入原生绘制模块，增加工程复杂度

---

## 方案 C（极端达标）
### 全部 UI 原生化

**核心思路**：
- 放弃 WebView2，全部 UI 改为原生渲染

**效果**：
- 100% 达标

**代价**：
- 重构成本极高，几乎等于重写前端

---

## 补充优化（配合使用）
即使走方案 A，也建议做以下优化以进一步压缩峰值：

1. **减少 WebView2 子进程启动因素**
   - 避免频繁创建/销毁窗口
   - 禁用不必要的 WebView2 功能模块

2. **前端内存优化**
   - 降低 DOM 复杂度
   - 图表实例及时销毁
   - 使用虚拟列表

3. **减少系统级采集子进程**
   - 替换 PowerShell GPU 采集（避免常驻 80MB）

---

## 推荐路径
**若目标是“必须 70%”**，建议直接选择 **方案 A（单 WebView 模式）**，其余方案只能部分缓解。
