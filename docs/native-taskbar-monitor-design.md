# 原生任务栏监测增量开发设计文档

## 1. 背景与目标

当前项目的任务栏监测窗口由 `taskbar.html` + Vue/Tauri WebView 构成。该窗口功能完整，但会额外引入一个长期存在的 WebView2 控件与对应子进程组。现阶段目标不是替换现有实现，而是在 **不破坏现有 WebView 任务栏监测功能** 的前提下，增量开发一套 **原生任务栏监测窗口**，并在主界面设置页新增一个独立开关用于启动该原生窗口。

本次设计遵循以下边界：

- 保留现有 `taskbarMonitorEnabled` 逻辑与 WebView 任务栏窗口能力。
- 新增一个单独开关，仅用于原生任务栏窗口试验性启停。
- 文档中的功能来源完全以当前 `taskbar` 页面和相关设置、事件、窗口逻辑为准。
- 原生任务栏第一阶段优先保证行为和数据一致，视觉允许接近但不要求像素级复刻。

## 2. 当前任务栏监测功能清单

### 2.1 窗口生命周期

当前 WebView 任务栏监测由 `useWindowStore().openTaskbarMonitor()` 创建，关闭时调用 `closeTaskbarMonitor()`：

- 任务栏窗口标签固定为 `taskbar`
- 入口页面为 `taskbar.html`
- 默认宽度 `520`
- 高度依据系统任务栏高度动态计算，最小为 `24`
- 默认无边框、透明、跳过任务栏、不可缩放、不可最小化
- 根据 `taskbarAlwaysOnTop` 决定初始置顶状态
- 若已存在同名窗口，则只执行 `show()`，不会重复创建

### 2.2 任务栏窗口定位行为

当前窗口位置来自 `api.getTaskbarInfo()` 返回的任务栏区域：

- 支持识别任务栏停靠边：上、下、左、右
- 依据当前主显示器缩放因子换算逻辑坐标
- 右下任务栏场景下将窗口定位到靠右留 `8px` 边距
- 左右停靠场景也会做相应偏移
- 页面挂载后会根据实际内容高度再次触发自适应 resize

### 2.3 展示内容

当前任务栏监测通过 `TelemetrySnapshot` 生成分段文本，支持按偏好开关显示以下项：

- CPU 占用
- CPU 频率
- CPU 温度
- GPU 占用
- GPU 温度
- 内存占用
- 应用自身 CPU / 内存占用
- 下载速率
- 上传速率
- 网络延迟

具体展示规则：

- `CPU`、`GPU`、`RAM`、`APP` 使用文本标签
- 下载与上传使用图标标签 `arrow_downward` / `arrow_upward`
- 网络速率显示单位固定为 `MB/s`
- 延迟为空时显示本地化的 `N/A`
- 当 `overlayPrefs.showWarning` 为开启时，CPU/GPU/RAM 数值根据阈值切换 `normal / warning / danger` 视觉等级
- `twoLineMode` 开启时按两行网格排布；关闭时按单行带分隔符排布

### 2.4 视觉模式

当前任务栏窗口支持以下视觉配置：

- `backgroundMode = transparent`
- `backgroundMode = dark`
- `backgroundMode = light`
- `twoLineMode = true/false`

这些偏好由 `useTaskbarPrefs()` 管理，并持久化到 `storageKeys.taskbarPrefs`。

### 2.5 交互行为

当前任务栏窗口支持以下交互：

- 左键按住拖拽窗口
- 双击显示主窗口
- 右键打开原生上下文菜单
- 内容高度变化时自动调整窗口大小
- 若开启“记住位置”，则移动后保存位置
- 若开启“锁定当前位置”，拖拽被禁止，窗口被强制恢复到锁定点

### 2.6 右键菜单功能

当前右键菜单来源于 `buildTaskbarMenuItems()`，功能包括：

- 显示主窗口 / 隐藏主窗口
- 总在最前
- 全屏时自动隐藏
- 记住位置
- 锁定当前位置 / 解锁位置移动
- 两行模式
- 透明主题
- 暗色主题
- 明亮主题
- 显示 CPU
- 显示 GPU
- 显示内存
- 显示 APP
- 显示下载
- 显示上传
- 关闭任务栏监测
- 退出应用

说明：

- 右键菜单中的“隐藏主窗口”当前调用的是 `closeMainWindow()`，会关闭 `main` WebView，而不是简单 `hide`
- “关闭任务栏监测”会关闭当前 `taskbar` 窗口，并同步关闭相关设置开关

### 2.7 置顶行为

当前任务栏窗口的置顶逻辑由以下两层组成：

- 显式设置：通过 `taskbarAlwaysOnTop` 决定目标状态
- 守护机制：`useTopmostGuard()` 周期性修复 TOPMOST 状态，并在失焦时再次修复

这意味着当前窗口不是一次性 `alwaysOnTop` 设置，而是具有“保活式置顶”行为。

### 2.8 全屏自动隐藏

当前存在两套相关机制：

1. `windowStore.startTaskbarFullscreenMonitor()`：从主窗口侧轮询全屏状态，必要时关闭或重新创建 `taskbar` WebView 窗口
2. `useFullscreenAutoHide()`：从 `taskbar` 页面自身轮询全屏状态，执行当前窗口 `hide/show` 并调整置顶守护

从当前代码行为看，WebView 任务栏监测在“全屏自动隐藏”场景下具备以下能力：

- 检测系统是否存在全屏前台窗口
- 隐藏任务栏监测窗口
- 退出全屏后恢复显示
- 恢复后重新应用置顶状态

### 2.9 提醒屏协同

当前任务提醒全屏窗口通过 `reminder://fullscreen` 同步事件与任务栏窗口协同：

- 提醒屏激活时，任务栏窗口暂停置顶守护
- 提醒屏激活时，任务栏窗口主动取消置顶
- 提醒屏退出后，任务栏窗口恢复原本置顶状态与守护逻辑

这是当前任务栏监测必须兼容的一个特殊场景。

### 2.10 配置与状态来源

当前任务栏监测涉及两类配置：

#### A. 全局设置 `AppSettings`

- `taskbarMonitorEnabled`
- `taskbarAlwaysOnTop`
- `taskbarAutoHideOnFullscreen`
- `taskbarPositionLocked`
- `rememberOverlayPosition`
- `language`

#### B. 任务栏偏好 `TaskbarPrefs`

- `showCpu`
- `showCpuFreq`
- `showCpuTemp`
- `showGpu`
- `showGpuTemp`
- `showMemory`
- `showApp`
- `showDown`
- `showUp`
- `showLatency`
- `twoLineMode`
- `backgroundMode`

### 2.11 多语言

当前菜单和任务栏标签具备中英文双语能力，来源于 `vue-i18n` 的 `overlay.*` 文案。

原生任务栏开发时至少需要保证：

- 标签内容与当前语言同步
- 右键菜单文案与当前语言同步
- 当语言设置变化时，原生任务栏能实时刷新显示文案

## 3. 本次增量开发的产品方案

### 3.1 新增开关

按照本次任务要求，不替换现有 `taskbarMonitorEnabled`，而是新增一个独立设置项：

- 中文文案建议：`原生任务栏监测（实验）`
- 英文文案建议：`Native Taskbar Monitor (Experimental)`

行为定义：

- 开关关闭：不启动原生任务栏窗口
- 开关开启：启动原生任务栏窗口
- 该开关与现有 `taskbarMonitorEnabled` 并存
- 第一阶段不强制关闭现有 WebView 任务栏监测，便于可行性测试与对比

### 3.2 功能对齐范围

原生任务栏第一阶段需尽量对齐以下现有能力：

- 显示实时监测指标
- 支持显示主窗口
- 支持隐藏主窗口
- 支持总在最前
- 支持全屏时自动隐藏
- 支持任务提醒时让出置顶
- 支持跟随语言切换
- 支持根据任务栏停靠边进行初始定位
- 支持关闭原生任务栏监测
- 支持退出应用

### 3.3 第一阶段允许降级的能力

为了保证增量开发速度，第一阶段允许以下能力先做接近实现：

- 视觉效果允许与 WebView 版接近，但不要求完全一致
- 两行模式可先做基础版
- 主题可先支持透明 / 暗色 / 明亮的基础样式，不强求 Web 版所有细节
- 位置记忆与锁定位置可在原型阶段延后到第二批补齐
- 拖拽逻辑可先支持基础拖拽，再补充边界修正与锁定恢复

## 4. 技术设计

### 4.1 总体思路

新增一个 Rust 原生窗口模块，由 Rust 直接创建 Win32 原生无边框透明窗口，用 GDI / Direct2D 风格的轻量绘制展示任务栏监测内容。

本次不删除现有 `taskbar` WebView 实现，而是新增一条独立链路：

- 前端设置页新增“原生任务栏监测”开关
- 开关变更后通过 Tauri command 通知 Rust
- Rust 根据当前状态创建、销毁、刷新原生任务栏窗口
- 原生任务栏所需数据直接复用当前 Rust 侧采样结果，不再引入额外 WebView

### 4.2 数据流

建议数据流如下：

1. Rust 现有 telemetry loop 继续采集系统快照
2. 在 Rust 侧新增 `NativeTaskbarViewModel`
3. 每次采样后若原生任务栏窗口开启，则将快照映射为轻量视图模型
4. 原生窗口只消费该视图模型进行绘制与状态更新

视图模型建议包含：

- 当前语言
- 当前主题模式
- 是否双行
- 每个 segment 的 label / value / extra / warningLevel / visible
- 当前置顶状态
- 当前是否因全屏隐藏
- 当前任务栏停靠边和目标位置

### 4.3 配置模型扩展

新增全局设置字段：

- `nativeTaskbarMonitorEnabled: boolean`

要求：

- 前端 `AppSettings` 增加字段
- Rust `AppSettingsPayload` 对应增加字段
- 默认值为 `false`
- 参与持久化、初始化、设置同步、导入导出

### 4.4 原生窗口行为设计

原生任务栏窗口建议具备以下行为：

- 无边框
- 透明背景
- 跳过任务栏
- 可选置顶
- 鼠标双击显示主窗口
- 右键弹出原生菜单
- 根据当前 taskbar edge 定位
- 依据文本内容计算窗口宽高
- 全屏时隐藏 / 恢复
- 接收提醒屏同步事件并暂时取消置顶

### 4.5 右键菜单对齐策略

第一阶段原生任务栏右键菜单与当前 WebView 版保持同一套菜单语义，菜单源建议直接复刻当前功能结构，而不是完全复用前端构建逻辑。

Rust 菜单至少覆盖：

- 显示主窗口 / 隐藏主窗口
- 总在最前
- 全屏时自动隐藏
- 两行模式
- 透明 / 暗色 / 明亮主题
- 显示 CPU / GPU / 内存 / APP / 下载 / 上传
- 关闭原生任务栏监测
- 退出应用

第二阶段再补齐：

- 记住位置
- 锁定当前位置 / 解锁位置移动
- CPU 频率 / 温度、GPU 温度、延迟等更细粒度控制菜单

## 5. 与现有代码的集成方案

### 5.1 前端改动

前端需要做的最小改动：

1. `AppSettings` 增加 `nativeTaskbarMonitorEnabled`
2. 默认设置、解析、持久化增加该字段
3. 设置页 `OverlayConfigSystemSection.vue` 新增独立开关
4. `settingsStore` 增加 `setNativeTaskbarMonitorEnabled()`
5. `app store` 暴露对应 setter
6. 主窗口 `bootstrap()` 时监听该设置变化并触发原生任务栏启停同步

### 5.2 Rust 改动

Rust 需要新增：

1. 原生任务栏状态管理模块
2. Win32 原生窗口创建与消息循环处理模块
3. 原生任务栏绘制模块
4. 原生任务栏命令接口：开启、关闭、刷新、应用设置
5. telemetry loop 到原生任务栏视图模型的映射逻辑

### 5.3 与现有 WebView 任务栏共存策略

当前阶段按用户要求，原生任务栏与 WebView 任务栏监测允许共存。

这意味着需要保证：

- 打开原生任务栏时，不影响 `taskbarMonitorEnabled`
- 关闭原生任务栏时，不影响 `taskbar` WebView 窗口逻辑
- 两者共享同一份数据采样源，但不能互相抢占设置持久化

## 6. 实施阶段划分

### 阶段 1：配置与空壳窗口

- 新增 `nativeTaskbarMonitorEnabled`
- 设置页新增开关
- Rust 新增启停命令
- 打开开关后创建原生空壳窗口
- 支持关闭开关后销毁窗口

验收标准：

- 开关可持久化
- 重启应用后能恢复开关状态
- 开关开启后出现原生窗口
- 关闭后窗口销毁

### 阶段 2：数据展示

- 将当前 telemetry 映射成原生文本段
- 先实现单行布局
- 支持 CPU / GPU / RAM / APP / 下载 / 上传 / 延迟 展示
- 支持基础主题色

验收标准：

- 原生任务栏能持续刷新数据
- 数据值与 WebView 版大体一致

### 阶段 3：交互与系统行为

- 双击显示主窗口
- 右键菜单
- 总在最前
- 全屏自动隐藏
- 提醒屏让出置顶

验收标准：

- 与现有 WebView 版核心交互一致

### 阶段 4：补齐高级能力

- 两行模式
- 位置记忆
- 锁定当前位置
- 更完整的细粒度显示控制

## 7. 风险与注意事项

### 7.1 透明绘制风险

Win32 原生透明窗口的绘制与文本抗锯齿可能与 WebView 版有差异，需要接受第一阶段视觉不完全一致。

### 7.2 多显示器与缩放

当前 WebView 版依赖 Tauri API 的缩放因子换算；原生版实现时必须重新验证：

- 多显示器
- 非 100% 缩放
- 任务栏在左/右/上/下

### 7.3 设置同步边界

当前很多设置同步是通过前端 `emitTo` 和本地存储完成；原生窗口不应依赖这些前端事件作为核心路径，应直接读取 Rust 管理的设置快照。

### 7.4 功能对齐优先级

本次的核心目标是验证“用原生窗口替代 WebView 任务栏是否可行并能减少 WebView2 负担”，因此应优先保证：

- 窗口可稳定创建/销毁
- 数据能稳定刷新
- 核心交互可用

而不是先追求完整视觉复刻。

## 8. 本次开发的明确结论

基于当前项目代码，增量开发“原生任务栏监测”是合理且可落地的。建议本轮直接按以下范围推进：

- 新增 `nativeTaskbarMonitorEnabled` 开关
- 在设置页中提供独立入口
- 新增 Rust 原生任务栏窗口原型
- 先实现数据展示、显示/隐藏主窗口、退出应用、置顶、全屏自动隐藏这几项核心能力
- 保留现有 WebView 任务栏监测，作为回退和对照基线

以上方案完整来源于当前 `taskbar` WebView 监测的现有功能，不包含额外扩展需求。
