硬件评分权重评估报告（结合当前项目实现）

日期：2026-03-06
范围：工具箱-硬件分析（前端启发式评分）

1. 现状与可用数据
当前项目能稳定获取的数据包括：
- CPU 型号、最高频率或当前频率
- GPU 型号、显存总量（VRAM）
- RAM 规格字符串与总量
- 硬盘型号列表（用于类型识别：NVMe/SSD/HDD）

评分实现位置：
- 评分与规则：src/components/toolkit/hardware/hardwareScoring.ts
- 组合与展示：src/components/toolkit/ToolkitHardwareTab.vue

2. 当前实现的权重摘要（实际代码）
2.1 基础评分
- CPU：频率线性映射 + 型号关键字加减分
- GPU：型号表评分与 VRAM 评分 70/30 混合
- RAM：容量分档为主 + 频率小幅加分
- Disk：NVMe=100，SSD=80，HDD=40，未知=60

2.2 场景评分
- Game：GPU 70% + CPU 20% + RAM 10%
- LLM：VRAM 50% + RAM 25% + CPU 15% + Disk 10%（HDD 额外 -15）
- Productivity：CPU 40% + RAM 30% + Disk 20% + GPU 10%
- Coding：仅基于 RAM 分档（HDD -10）
- Balance：CPU/GPU/RAM 标准差惩罚 + HDD -10

2.3 总分
总分 = Balance 15% + Game 20% + LLM 20% + Overall 20% + Productivity 15% + Coding 10%
其中 Overall = CPU 35% + GPU 35% + RAM 20% + Disk 10%

3. 合理性评估（结合当前数据约束）
结论：当前权重整体可用，但存在场景与指标映射不一致、解释力偏弱的问题。
核心问题：
1. Coding 只看 RAM，忽略 CPU 与 Disk 对编译与 I/O 的影响，导致结果偏差大。
2. LLM 过度依赖 VRAM，但 GPU 架构与带宽差异未充分体现；Disk 权重偏高。
3. Overall 的 Disk 权重偏低，且 HDD/SSD 的体验差异未反映到综合权重层面。
4. Balance 的统计惩罚合理，但与总分权重叠加后对“短板”仍不够敏感。

4. 建议的“最合理权重”（不增加新数据前提）
目标：保持可解释性，同时让场景评分更符合真实体验。

4.1 建议调整的场景权重
Coding（建议替换）
- CPU 45% + RAM 35% + Disk 20%
理由：编译与索引高度依赖 CPU 与 I/O，RAM 影响并发与缓存。

LLM（建议替换）
- VRAM 55% + GPU_Model 25% + RAM 15% + Disk 5%
理由：VRAM 仍是“入场券”，但 GPU 架构与 RAM 影响不应被低估；Disk 仅影响加载。

Productivity（建议微调）
- CPU 35% + RAM 30% + Disk 20% + GPU 15%
理由：常见办公/创作场景对 GPU 的占比略高于当前实现。

Game（保持）
- GPU 70% + CPU 20% + RAM 10%
理由：当前权重与用户体验一致，稳定可解释。

Balance（保持）
- 标准差惩罚 + HDD 额外惩罚
理由：在现有数据粒度下是合理的短板度量。

4.2 建议调整的总分组合权重
当前：Balance 15% + Game 20% + LLM 20% + Overall 20% + Productivity 15% + Coding 10%
建议：
- Balance 18% + Game 18% + LLM 18% + Overall 20% + Productivity 16% + Coding 10%
理由：提升 Balance 权重以放大短板影响，均衡多场景权重避免单项极端。

4.3 建议调整的 Overall 权重
当前：CPU 35% + GPU 35% + RAM 20% + Disk 10%
建议：
- CPU 32% + GPU 32% + RAM 22% + Disk 14%
理由：在现代系统中磁盘对体感影响更大，适度上调 Disk 权重。

6. 可选改进（不影响权重，但提升可靠性）
- CPU 引入核心数或线程数（若可取）
- GPU 型号表定期更新并区分移动端 SKU
- Disk 识别加强：加入 NVMe Gen3/Gen4 细分与 SATA SSD 区分

总结
在当前数据约束下，上述权重能更贴近实际体验，同时保持前端可解释性与实现成本可控。
