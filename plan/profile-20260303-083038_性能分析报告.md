PulseCoreLite 性能分析报告（profile-20260303-083038）

一、报告信息

1. 数据文件
   - d:\PROJECT\PulseCore\release\profile-20260303-083038.jsonl
2. 采样规模
   - 共 300 条样本
   - 刷新频率 1000ms

二、关键指标概览

1. 系统资源
   - 系统 CPU 平均 38.43%，峰值 48.86%
   - 系统内存平均 79.20%，峰值 82.71%
2. 应用总体
   - appCpuUsagePct 平均 0.268，峰值 0.782
   - appMemoryMb 平均 614.02MB，峰值 723.91MB
3. 进程结构特征
   - msedgewebview2.exe 记录数 2388（约每秒 8 个实例）
   - powershell.exe 记录数 311（高频出现）
   - conhost.exe 记录数 310（与子进程链路相关）
4. 单进程峰值
   - msedgewebview2.exe 内存峰值约 167.85MB
   - powershell.exe CPU 峰值约 220.31%

三、占用过高的主要原因

1. WebView 进程常驻数量偏多，造成内存长期高位
   - 多个 msedgewebview2.exe 常驻，单实例内存不低
   - 实例数叠加后成为主要内存来源
2. PowerShell 子进程触发频繁，造成 CPU 尖峰明显
   - 观察到多次 200%+ CPU 峰值
   - 说明存在并发脚本任务或高频轮询任务
3. 系统基础压力偏高，放大卡顿风险
   - 系统内存长期在 79%~83%
   - C 盘容量使用率约 91%，空间紧张时更易出现分页和 IO 抖动

四、优化方案（按优先级）

1. P0：子进程调度与并发治理
   - 给 PowerShell 任务增加并发上限与任务去重
   - 同类任务使用单飞锁，避免同一周期重复执行
   - 把高频轮询改为事件触发或分层采样
2. P0：WebView 生命周期优化
   - 复用 WebView 实例，减少重复创建
   - 不可见窗口降频（timer、动画、轮询）
   - 关闭无活动页面时主动释放资源
3. P1：渲染与内存优化
   - 大列表虚拟化、图片懒加载、缓存上限控制
   - 降低跨进程高频通信和大对象序列化
   - 清理未释放的订阅、定时器和临时对象引用
4. P1：系统与部署侧优化
   - 清理 C 盘空间，建议保留至少 15%~20% 空闲
   - 高频日志与临时文件迁移到非系统盘并启用滚动清理

五、建议的落地顺序

1. 第一阶段（优先见效）
   - 先做子进程并发限制与任务去重
   - 同步做 WebView 复用和不可见降频
2. 第二阶段（稳定收敛）
   - 做前端渲染减负与内存泄漏治理
   - 做日志与临时文件策略调整
3. 第三阶段（持续防回归）
   - 增加监控指标：活跃 WebView 数、每秒子进程创建数、任务队列长度
   - 建立性能回归门槛并固化到发布流程

六、建议验收指标

1. 资源目标
   - appMemoryMb 峰值较当前下降 15% 以上
   - powershell.exe CPU 峰值较当前下降 40% 以上
2. 稳定性目标
   - 系统内存占用曲线波动幅度下降
   - UI 刷新卡顿与短时掉帧显著减少

七、结合当前代码的落地技术方案（可直接排期）

1. 当前采集与渲染链路（现状）
   - 后端主循环：`src-tauri/src/app.rs` 的 `start_telemetry_loop` 按 `refresh_rate_ms` tick，读取可见窗口后执行 `state.collect_snapshot(current_rate)`，再按窗口 `emit_to(..., "telemetry://snapshot", ...)` 推送。
   - 采集器核心：`src-tauri/src/core/collectors/system_collector.rs` 的 `collect` 每轮刷新 CPU/内存/网络，并在分层间隔下刷新 GPU、磁盘、应用进程占用。
   - 前端接收：`src/stores/app.ts` 在 `bindEvents` 监听 `telemetry://snapshot`，调用 `telemetryStore.pushSnapshot`。
   - 状态落地：`src/stores/telemetryStore.ts` 使用 `markRaw` 替换 `snapshot`，页面与 taskbar 依赖该快照的 computed 自动重算。
   - UI 渲染：`src/pages/index.vue` 通过 `useOverlayMetrics` 映射指标，`src/pages/taskbar.vue` 通过 `segments` 组装文本并渲染。

2. P0（第一阶段，优先收益最高）
   - 空窗降频/暂停采样
     - 问题：`start_telemetry_loop` 即使无可见消费者也会持续 `collect_snapshot`。
     - 改造：引入两级频率（active/idle），当 `visible_labels` 为空时切换到 idle（建议 2000~5000ms），并允许在无 profiling 任务时直接跳过昂贵采样。
     - 预期：空闲 CPU 降低、后台功耗下降。
   - GPU 采样子进程治理
     - 问题：`query_gpu_metrics_windows` 每次采样拉起 `powershell.exe`，在高频率下形成 CPU 尖峰。
     - 改造：加入单飞锁 + 连续失败退避（如 1s→2s→5s），并限制最小 GPU 采样间隔到 1000ms；保留 DXGI 内存兜底。
     - 预期：`powershell.exe` 峰值与出现频次显著下降。
   - 主窗口标题刷新降频
     - 问题：`start_telemetry_loop` 对 main 窗口每 tick `set_title`。
     - 改造：仅在文本变化时更新，且设置最小间隔（建议 1000ms）。
     - 预期：减少 IPC 与窗口系统调用抖动。

3. P1（第二阶段，稳定收敛）
   - 应用进程树采样降本
     - 问题：`collect_process_tree_pids_windows` + `query_process_cpu_and_memory_windows` 在 app usage 采样路径上频繁全量遍历。
     - 改造：进程树缓存 3~5 秒，采样窗口不足时复用上次结果；仅对存活 PID 增量更新。
     - 预期：降低 WinAPI 枚举与句柄开销。
   - 前端渲染分层与节流
     - 问题：每次 snapshot 替换会触发 overlay/taskbar 全链路 computed 重算。
     - 改造：在 `useOverlayMetrics` 与 taskbar `segments` 增加低变化字段复用策略（如字符串缓存、仅数值变化超阈值时重建段）。
     - 预期：降低前端重排与文本测量频率，减少短时掉帧。

4. 验收口径（与本项目实现对齐）
   - 采样链路：`telemetry://snapshot` 事件速率在 active/idle 模式下符合配置预期。
   - 子进程链路：`powershell.exe` 每分钟创建次数较当前基线下降 50% 以上。
   - 资源指标：appMemoryMb 峰值下降 15%+，并保持 UI 功能无退化。
   - 体验指标：overlay/taskbar 在 10 分钟连续运行下无明显卡顿峰值。
